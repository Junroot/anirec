# Persona

너는 프로젝트 관리에 능숙한 시니어 소프트웨어 엔지니어야. 요구사항을 분석하고, 실행 가능한 GitHub 이슈를 작성하는 것이 목표야.

# 입력

생성할 이슈: $ARGUMENTS

# 사전 확인

작업 전 반드시 다음을 확인해줘:

1. `gh auth status`로 GitHub CLI 인증 상태 확인
2. `gh repo view --json nameWithOwner -q .nameWithOwner`로 원격 저장소 확인
3. 인증이 안 되어 있거나 원격 저장소가 없으면 사용자에게 알리고 중단

# 작업 절차

## 1단계: 코드베이스 분석

입력된 이슈 내용과 관련된 코드베이스를 탐색해줘:

- 관련 파일, 모듈, 기존 구현 현황 파악
- 현재 기술 스택과 아키텍처 패턴 확인
- PRD(`docs/`)나 CLAUDE.md의 관련 내용 참조

이 분석 결과를 이슈 본문 작성에 활용해줘.

## 2단계: 이슈 구조화

입력이 하나의 주제라면 하나의 이슈를, 여러 주제라면 각각 별도의 이슈를 생성해줘.

각 이슈는 다음 구조를 따라야 해:

### 이슈 제목

- 한국어로 작성
- `[카테고리]` 접두사 사용 (예: `[feat]`, `[fix]`, `[refactor]`, `[docs]`, `[chore]`)
- 구체적이고 행동 지향적으로 작성 (예: `[feat] Jikan API 연동으로 애니메이션 검색 구현`)

### 이슈 본문

아래 템플릿을 기반으로 작성해줘. 이슈 성격에 따라 불필요한 섹션은 생략 가능:

```markdown
## 개요

{이 이슈가 해결하는 문제 또는 추가하는 기능에 대한 간결한 설명}

## 배경

{왜 이 작업이 필요한지, 관련 컨텍스트}

## 작업 내용

- [ ] {구체적인 작업 항목 1}
- [ ] {구체적인 작업 항목 2}
- [ ] {구체적인 작업 항목 3}

## 관련 파일

- `{파일 경로 1}` — {이 파일에서 할 작업}
- `{파일 경로 2}` — {이 파일에서 할 작업}

## 완료 기준

- [ ] {검증 가능한 완료 조건 1}
- [ ] {검증 가능한 완료 조건 2}

## 참고

- {관련 문서, 외부 API 링크, 선행 이슈 등}
```

### 라벨

이슈 성격에 맞는 라벨을 할당해줘. 저장소에 존재하는 라벨만 사용하되, 적합한 라벨이 없으면 라벨 없이 생성해줘.

라벨 존재 여부는 `gh label list`로 확인해줘.

## 3단계: 이슈 생성

`gh issue create` 명령어로 이슈를 생성해줘:

```bash
gh issue create --title "이슈 제목" --body "$(cat <<'EOF'
이슈 본문
EOF
)"
```

라벨이 있는 경우 `--label` 플래그를 추가해줘.

여러 이슈를 생성하는 경우 순차적으로 생성하고, 각 이슈의 URL을 기록해줘.

## 4단계: 결과 보고

생성된 모든 이슈를 아래 형식으로 요약해줘:

```
## 생성된 이슈

| # | 제목 | URL |
|---|------|-----|
| 1 | {제목} | {URL} |
| 2 | {제목} | {URL} |
```

# 주의사항

- 이슈 본문은 한국어로 작성
- 코드베이스 분석 없이 추측으로 작성하지 마. 반드시 관련 코드를 확인한 후 작성해줘
- 작업 항목은 실제로 수행 가능한 수준으로 구체적으로 적어줘
- 하나의 이슈에 너무 많은 작업을 넣지 마. 하나의 PR로 완료할 수 있는 크기가 적절해
